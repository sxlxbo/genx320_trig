# 分析端重建计划

> 本文记录第一版 `rgb-sync-v1` 分析端的设计依据与实现过程。针对采集端
> `rgb-black-sync-v2` 的后续改造，见
> [分析端第二轮重建计划.md](./分析端第二轮重建计划.md)。第二轮完成前，
> 本文中的连续 `R→G→B`、`RGB_RISE/color` 和 50 个 RGB 周期等约定仅适用于
> 第一版历史数据，不能直接用于新版样本。

## 1. 目标与交付结果

分析端读取采集端生成的 `genx320_led_sync_*.raw`，优先同时读取同名 CSV 元数据，按外部 Trigger 将事件流切分为 R、G、B 三类照明响应，最终输出一张等分为四个区域的 PNG：

```text
┌──────────────────┬──────────────────┐
│ R 单通道重建图   │ G 单通道重建图   │
├──────────────────┼──────────────────┤
│ B 单通道重建图   │ RGB 合成结果     │
└──────────────────┴──────────────────┘
```

四个区域必须具有完全相同的像素尺寸。前三格使用灰度显示对应通道的物体响应，最后一格显示三通道合成结果；每格仅允许在不改变区域尺寸的前提下增加标题。默认输出原始传感器分辨率的 2×2 拼图，即 GenX320 常见的 `640×640` 像素；程序实际应调用 `RawReader.get_size()`，不能把 `320×320` 写死。

计划交付的分析程序暂定名为 `reconstruct_rgb.py`，推荐命令行形式：

```bash
python reconstruct_rgb.py input.raw --metadata input.csv --output result.png
```

除最终 PNG 外，再保存一份同名 `*.report.json`，记录触发数量、有效 RGB 周期数、事件数量、时间偏移、丢弃周期、归一化参数和警告，保证结果可追溯。

## 2. 对现有采集规则的确认

分析端必须严格遵守 `caiji/sync_protocol.py`、`caiji/sync_led_trig.py` 和 `caiji/receive_record.py` 已实现的协议：

- 协议版本为 `rgb-sync-v1`，外部 Trigger 通道为 `id=0`。
- PREAMBLE 为 3 个脉冲；接收端确认 PREAMBLE 后才开始写 RAW，因此最终 RAW 通常不含完整 PREAMBLE。
- RAW 内随后有 2 个 START 脉冲、保护期、正式 RGB 段、保护期及 4 个 STOP 脉冲。
- 正式段为 `R → G → B` 循环，首个正式颜色固定为 R。
- 相邻正式 Trigger 的标称间隔为 `100000 µs`，容差沿用采集端的 `±20000 µs`。
- 默认时长 15 s，共 150 次颜色切换，即约 50 个完整 RGB 周期。
- LED 先执行 `pixels.show()`，函数返回后再发出 2 ms Trigger。Trigger 是“颜色已提交”的标志，但光学变化相对 Trigger 的真实偏移仍需从 RAW 测量。
- 同名 CSV 中 `role=RGB_RISE` 的行已经给出 `timestamp_us`、`cycle_index` 和 `color`，应作为首选分段依据；CSV 缺失时才从 RAW 的 Trigger 序列恢复协议。

目前工作区中没有实际 `.raw` 或对应 `.csv` 样本，因此本计划能确定软件结构和算法，但事件响应窗、噪声阈值、色彩增益等数值必须在拿到真实文件后标定，不能仅凭代码臆定。

## 3. 成像原理与可达到的结果边界

GenX320 输出事件 `(x, y, p, t)`：当某像素的对数亮度变化超过阈值时产生事件，`p` 表示变亮或变暗。它不直接输出每个像素的绝对亮度，也不是带 Bayer 阵列的普通 RGB 相机。

对静止场景，当前采集的三种主要转换为：

```text
B → R：dR ≈ log(IR) - log(IB)
R → G：dG ≈ log(IG) - log(IR)
G → B：dB ≈ log(IB) - log(IG)
```

其中每个差值可由该转换附近的正、负事件数近似。三个方程之和理论上为零，所以它们只能稳定提供相对色度，不能单独确定绝对亮度。RAW 中第一次 `黑 → R` 转换可以提供一帧亮度/轮廓锚点，但它只有一次，信噪比可能低于后续 49 次可重复转换。

因此结果分两级定义：

1. **第一阶段可交付结果**：无需训练数据，以事件响应重建 R/G/B 通道物体轮廓和纹理，并生成一致、可解释的 RGB 伪彩结果。它应能看出物体，但颜色和明暗不承诺与普通 RGB 相机完全一致。
2. **标定后结果**：使用色卡/灰卡参考数据估计通道增益、3×3 色彩矩阵和 gamma，可获得较可信的 RGB 颜色。

相关研究已经证明“单色 DVS + 主动 RGB 照明”可以重建全分辨率彩色图，但也指出单一照明强度不利于区分灰阶，多强度照明和 CNN 会显著提高颜色忠实度。现有协议只有一种 `SAFE_BRIGHTNESS=0.2`，所以第一版应优先做可解释的经典基线，不应直接宣称得到绝对真实色彩。

## 4. 总体处理流程

```text
RAW + 可选 CSV
      │
      ▼
文件/协议校验 ──失败──> 明确报错与诊断报告
      │
      ▼
流式解码 CD 事件和外部 Trigger
      │
      ▼
CSV 对齐或从 START/RGB/STOP 序列恢复颜色标签
      │
      ▼
估计 LED 光学响应相对 Trigger 的时间偏移
      │
      ▼
按转换切窗，生成正/负事件计数与多时间片伪帧
      │
      ▼
坏点、背景、运动异常过滤 + 50 周期稳健融合
      │
      ├──> R 单通道图
      ├──> G 单通道图
      ├──> B 单通道图
      └──> 约束求解/标定矩阵 → RGB 合成图
                                  │
                                  ▼
                       2×2 等分 PNG + JSON 报告
```

## 5. 分阶段实施方案

### 阶段 A：RAW 解码、元数据读取和协议校验

1. 使用 Metavision SDK 的 `RawReader` 流式读取，调用 `load_delta_t()` 获取 CD 事件，并通过 `get_ext_trigger_events()` 获取外部 Trigger；每次处理后清空已读取的 Trigger 缓存。
2. 通过 `get_size()` 获取 `(height, width)`，创建对应尺寸的计数数组。
3. 若存在 CSV：
   - 校验必要列 `timestamp_us,id,polarity,role,cycle_index,color`；
   - 只采用 `id=0`、`polarity=1`、`role=RGB_RISE`；
   - 检查 `cycle_index` 连续且 `color == (R,G,B)[index % 3]`；
   - 检查 CSV 时间戳能在 RAW Trigger 中以小容差匹配。
4. 若 CSV 缺失：复用采集端 `verify_raw()` 的判定思想，在 RAW 上升沿中识别 30 ms START 对、首个正式脉冲、100 ms 正式序列和 30 ms STOP 序列；绝不能简单按“第一个 Trigger 是 R”处理。
5. 默认至少需要 3 个完整 RGB 周期才允许重建；不足时失败退出。少于预期 50 周期但仍满足最低数量时给出降质警告。
6. 不一次性把整个 RAW 载入内存。第一遍提取 Trigger 和全局事件率，第二遍按已确定的窗口累计特征，便于处理更长录制。

### 阶段 B：确定真实光学响应窗口

固定地使用 `[trigger, trigger+100 ms)` 会有两个问题：LED 光学变化可能略早于 GPIO 上升沿，且运动/背景事件会污染整个 100 ms 区间。因此采用数据驱动窗口：

1. 对每个正式 Trigger 前后，例如 `[-5 ms, +50 ms]`，以 0.5 ms 或 1 ms 分箱统计全画面事件率。
2. 将所有周期按相对 Trigger 时间叠加，寻找稳定的事件率峰值，得到 `optical_offset_us`。
3. 初始响应窗口建议以峰值为中心覆盖主要能量，例如累计包含 90% 转换事件的最短窗口，并限制在 `[-5 ms, +40 ms]` 内；实际参数写入报告。
4. 窗口上界不能越过相邻 Trigger 的中点。
5. 从每个响应窗末尾选择等长“安静窗”估计背景事件率，响应图减去按时长缩放的背景。
6. 如果峰值落在搜索边界、各颜色峰值偏移差异过大，或响应窗与下一个 Trigger 重叠，则停止正式重建并报告同步/LED 时序问题。

此阶段应额外输出一张可选的 `*_timing.png` 诊断图，显示 Trigger 对齐后的事件率曲线，方便确定 LED 和 Trigger 是否真正同步。

### 阶段 C：构建每次颜色转换的事件特征

对每次 `RGB_RISE` 建立下列特征，目标颜色由 CSV 或 `cycle_index % 3` 决定：

- `N_pos[y,x]`：响应窗口内正事件数；
- `N_neg[y,x]`：响应窗口内负事件数；
- `S[y,x] = N_pos - N_neg`：有符号对数亮度变化近似；
- `A[y,x] = N_pos + N_neg`：不依赖极性的物体边缘/响应强度；
- 可选多时间片张量：把响应窗分成 4～8 个时间片，保留脉冲响应衰减信息，供后续 CNN 使用。

预处理顺序：

1. 根据 START 后首个正式 Trigger 之前的安静段估计每像素热噪声。
2. 标记长期事件率异常的热像素，不用单次硬阈值误删真实高反射边缘。
3. 对正、负极性分别扣除背景期望，结果截断到非负。
4. 使用轻量的 3×3 中值滤波或带边缘保持的双边滤波去除椒盐噪声；滤波前保留原始特征供诊断。
5. 每个周期计算全局事件数、峰值时刻和空间重心。用中位数绝对偏差（MAD）剔除明显过曝、漏光、抖动或物体运动造成的异常周期。

### 阶段 D：跨周期稳健融合为三个通道

现有协议约有 50 次 R、50 次 G、50 次 B 响应，重复采集是提高信噪比的关键。对同一目标颜色的有效转换：

1. 分别堆叠 `S` 和 `A`。
2. 默认使用逐像素中位数或 10% 截尾均值，不直接求和，避免少数异常周期支配结果。
3. 对不同有效周期数量做归一化。
4. 用 `A` 形成可靠性/物体支持掩膜；低于背景统计阈值的像素置零。
5. 生成可见的灰度单通道图：以稳健融合后的响应强度做 `asinh` 或 `log1p` 压缩，再用 1%～99% 分位数映射到 8 位。三个通道共享归一化尺度，禁止各自独立拉满导致错误颜色。

第一版应同时实现两种可比较的通道定义：

- **响应图基线**：按目标颜色聚合 `A`，最稳健，适合先确认物体是否被重建，但表示的是“切换到该颜色时的变化强度”。
- **有符号差分图**：按目标颜色聚合 `S`，用于求解相对 RGB 色度，物理含义更明确，但对噪声和极性阈值差异更敏感。

报告中保存所选方法，CLI 用 `--method response|log-ls` 切换，默认先使用 `log-ls`，失败时明确降级为 `response`，不能静默改变算法。

### 阶段 E：RGB 约束重建

#### E1. 无训练、无色卡的经典基线

对每个像素，把稳健融合后的三种转换写成：

```text
lR - lB = dR
lG - lR = dG
lB - lG = dB
lR + lG + lB = 0    （消除不可观测的共同偏置）
```

其中 `lR,lG,lB` 是相对对数通道响应，`dR,dG,dB` 由有符号事件数乘以待标定的对比度阈值近似。用带权最小二乘求解，权重来自每个通道的响应可靠性。因为共同亮度不可观测，再使用以下亮度包络之一恢复物体形状：

1. 首选：第一次 `黑 → R` 的响应图，经空间滤波后作为亮度锚点；
2. 若首帧过噪：使用三个通道总事件强度形成公共亮度包络；
3. 对亮度包络进行单独的分位数归一化，再与相对色度相乘。

最后执行通道增益、裁剪、gamma 映射，得到 RGB。无色卡时仅输出“相对/伪彩 RGB”，JSON 中写入 `color_fidelity: uncalibrated`。

#### E2. 有色卡后的线性色彩标定

在相同距离、环境光、LED 亮度和相机 bias 下录制标准色卡与灰卡：

1. 手工或半自动选取色块 ROI；
2. 以每个 ROI 的重建 RGB 特征拟合白平衡增益和 3×3 颜色校正矩阵；
3. 用保留色块评估误差，避免只报训练误差；
4. 将参数保存为 `calibration.json`，包含设备、距离、LED 亮度、bias 和协议版本；
5. 分析时用 `--calibration calibration.json` 加载，条件不匹配时警告。

#### E3. 后续 CNN 升级

只有获得同视角普通 RGB 真值图和足量样本后再考虑 CNN。输入可以是每种转换的正/负极性多时间片张量，输出为 3 通道图。必须按“物体/场景”划分训练集和测试集，防止同一物体泄漏造成虚高结果。CNN 是第二阶段优化，不应阻塞经典基线交付。

### 阶段 F：四宫格成图

1. `panel_size = (width, height)`，创建 `2*width × 2*height` 的 RGB 画布。
2. R、G、B 单通道面板都用灰度复制到三个显示通道，避免用红/绿/蓝色染色掩盖真实强弱差异。
3. RGB 面板使用标定后或未标定的合成图。
4. 四格使用同样的 ROI、方向、几何变换和尺寸，不允许某格单独裁切或缩放。
5. 标题可画在每格内部固定高度的半透明条上，或另存无标题科研版；不能因标题使四格尺寸不等。
6. PNG 默认 8 位 sRGB；同时可选保存 16 位单通道 TIFF/PNG 和浮点 `features.npz`，方便后续算法比较。

## 7. 关键配置项

配置项不能散落在代码中，建议集中写入 YAML 并同步到报告：

```yaml
protocol:
  channel_id: 0
  rgb_interval_us: 100000
  rgb_tolerance_us: 20000
timing:
  search_pre_us: 5000
  search_post_us: 50000
  bin_us: 1000
  energy_fraction: 0.90
filter:
  hot_pixel_mad: 8.0
  cycle_outlier_mad: 5.0
  spatial_filter: median3
render:
  low_percentile: 1.0
  high_percentile: 99.0
  tone_map: asinh
```

这些是启动参数，不是最终标定值。真正阈值要由第一批 RAW 的事件率曲线、背景段和重复周期统计决定。

## 8. 测试与验收标准

### 8.1 单元测试

- 用合成 Trigger 序列验证 START、150 个 RGB、STOP 的解析。
- 插入漏脉冲、重复脉冲、错误通道、错误极性和周期越界，必须得到明确失败原因。
- 构造已知 `lR,lG,lB`，生成三组差分，验证约束最小二乘能恢复相对色度。
- 验证全部零事件、单个热像素和极端事件数不会产生除零、NaN 或全白图。
- 验证输出宽高严格为 `(2*width, 2*height)` 且四格数组尺寸一致。

### 8.2 实际 RAW 验收

对第一份真实录制至少输出以下统计：

- CSV 与 RAW Trigger 匹配率为 100%；
- 正式 RGB 上升沿数量与颜色顺序正确；
- 每种颜色至少保留 80% 周期，低于此值必须警告；
- 三种颜色的光学事件峰均落在相邻 Trigger 中点之前；
- 背景扣除后，目标区域事件密度显著高于安静背景；
- 四格方向一致、无镜像/转置错误，物体轮廓在三个通道中空间对齐；
- 同一 RAW 重复运行得到逐像素一致的 PNG 和一致报告参数。

### 8.3 色彩验收

- 无色卡版本只验收物体可见性、通道分离和可重复性，不以“肉眼颜色像普通相机”为硬指标。
- 标定版本使用未参与拟合的色块计算 RGB RMSE，并建议计算 CIEDE2000；阈值由实验用途确定。
- 至少比较三种结果：直接事件响应拼接、约束最小二乘、带色彩矩阵的结果，以确认每一级优化确实有效。

## 9. 风险、失败表现和处置

| 风险 | 结果表现 | 分析端处置 |
|---|---|---|
| RAW 没有同名 CSV | 颜色边界不明确 | 从 START/RGB/STOP 恢复；同时给出降级警告 |
| LED 光学变化早于 Trigger | Trigger 后窗口漏事件 | 搜索负偏移并使用 Trigger 前缓冲窗口 |
| 环境光过强 | RGB 切换事件很少 | 报告低信噪比；建议降低环境光或提高安全范围内 LED 亮度 |
| 物体/相机移动 | 100 ms 周期内出现非照明事件 | 通过峰外事件率和周期空间重心筛除；严重时拒绝静态重建 |
| 热像素/椒盐噪声 | 单通道大量亮点 | 背景统计坏点掩膜 + 轻量空间滤波 |
| 单强度照明 | 灰阶和真实色彩不准 | 明确标为未标定伪彩；采集端后续增加多亮度 |
| R/G/B LED 实际亮度不等 | 合成图偏色 | 灰卡白平衡和 3×3 色彩矩阵 |
| 现有转换是颜色到颜色 | 通道图含前一颜色影响 | 用有符号转换方程联合求解，不把目标色计数直接当绝对亮度 |
| 事件率控制器/抗闪烁设置影响响应 | 峰值被削弱或截断 | 报告相机 bias/AFK/ERC/STC 配置，并在标定和正式采集间保持一致 |

## 10. 对采集端的后续改进建议

这些改动不是第一版分析程序的前置条件，但会明显提高最终图像质量：

1. 每次颜色前插入黑场，并为黑场也发送 Trigger，例如 `黑→R→黑→G→黑→B`。这样每个通道都有独立的亮起和熄灭响应，不再完全依赖颜色间差分。
2. 每种颜色使用 3 个以上亮度级别，而不是只有 `SAFE_BRIGHTNESS=0.2`；这有助于恢复灰阶和绝对强度。
3. 在正式段开始前增加数个 `黑→R` 重复，而不是只保留一次亮度锚点。
4. 用光电二极管实测 LED 光输出与 GPIO Trigger 的偏移、抖动和衰减曲线，并把标定值写入 sidecar。
5. sidecar 增加协议版本、相机序列号、分辨率、bias、AFK/ERC/STC、LED 亮度、距离和环境光。
6. 增加灰卡/色卡专用采集模式，为每套硬件生成可复用的 `calibration.json`。

## 11. 实施顺序与里程碑

1. **M1：可读、可分段**——完成 RAW/CSV 读取、协议校验、事件率时序诊断。
2. **M2：物体可见**——完成响应窗、背景扣除、坏点处理和 R/G/B 灰度响应图。
3. **M3：四格交付**——完成相对对数 RGB 求解、共享归一化和严格等分的最终 PNG。
4. **M4：可验证**——完成 JSON 报告、合成数据单元测试和至少一份真实 RAW 的验收。
5. **M5：颜色可信**——完成灰卡/色卡采集、3×3 色彩标定及独立色块误差评估。
6. **M6：质量升级（可选）**——修改采集协议为多亮度/插黑帧，或积累数据后训练 CNN。

## 12. 调研依据

- [Metavision SDK Event I/O 文档](https://docs.prophesee.ai/stable/api/python/core/event_io.html)：`RawReader` 可读取 RAW、按时间片加载 CD 事件、读取外部 Trigger 并查询传感器尺寸。
- [Prophesee GenX320 产品资料](https://www.prophesee.ai/wp-content/uploads/2025/01/GENX320-Product-Brief-2025-CCAM-OK.pdf)：GenX320 为 320×320 对比度事件传感器，提供微秒级时间戳，并包含 AFK、ERC、STC 等会影响事件输出的功能。
- [Illumination-Based Color Reconstruction for the Dynamic Vision Sensor](https://arxiv.org/abs/2211.06695)：验证了单色 DVS 配合主动 RGB 闪烁进行全分辨率彩色重建的可行性，采用转换响应伪帧、线性估计和 CNN；同时说明单一照明强度会损害灰阶/颜色区分，多强度和空间相关性建模能改善结果。

## 13. 开始编码前所需输入

为了把本计划落实成可验证代码，至少需要一组采集端判定成功的：

```text
genx320_led_sync_时间.raw
genx320_led_sync_时间.csv
```

最好再提供对应实验照片或文字说明（物体颜色、位置、环境光、相机到物体距离），以及一段无物体/纯黑场录制。第一份 RAW 到位后，优先完成 M1 和 M2，先用时序诊断确认事件确实来自 RGB 切换，再决定 M3 的具体阈值和色彩标定路径。
