# GenX320 分析端第二轮重建计划

> 实施状态（2026-07-21）：M1～M3 已完成，v2 合成协议/时序/特征/重建测试
> 11 项全部通过；采集端 5 项协议回归同时通过。M4 等待首份
> `rgb-black-sync-v2` RAW/CSV。当前 `input/` 中的 2026-07-15 样本是
> `rgb-sync-v1`，不会被分析端静默当作 v2 使用。

## 1. 背景与本轮目标

采集端已经由 `rgb-sync-v1` 升级为 `rgb-black-sync-v2`。正式照明序列从连续的：

```text
R → G → B → R → ...
```

改为：

```text
BLACK → R → BLACK → G → BLACK → B → ...
```

默认每个状态保持 100 ms，执行 25 个完整 RGB 周期，共产生 150 个正式状态 Trigger，其中包含 75 个 `BLACK_RISE` 和 75 个 `COLOR_RISE`。每个颜色各有 25 次独立的 `BLACK→Color` 亮起响应。

第二轮分析端的目标不是简单兼容新的 CSV 列名，而是把内部数据模型、事件分窗和重建含义统一调整为“黑场基准的独立颜色响应”：

```text
BLACK → R  构建 R 主通道
BLACK → G  构建 G 主通道
BLACK → B  构建 B 主通道
R/G/B → BLACK  作为熄灭响应和质量诊断
```

最终仍输出 R、G、B 三张单通道图、RGB 合成图、时序诊断图和 JSON 报告。无灰卡或色卡标定时，结果继续标记为相对色彩或事件响应伪彩，不能表述为普通相机意义上的绝对 RGB。

## 2. 当前代码与 v2 协议的不一致

现有分析端是按 v1 实现的，主要不一致点如下：

| 模块 | 当前 v1 行为 | v2 所需行为 |
|---|---|---|
| `protocol_parser.py` | 只读取 `RGB_RISE` 和 `color`，要求 `cycle_index` 逐 Trigger 递增 | 读取 `protocol_version/state_index/cycle_index/phase_index/state/role`，校验六状态周期 |
| `timing.py` | 把所有 RGB Trigger 合并估计一个换色响应 | 分别统计 COLOR 亮起和 BLACK 熄灭响应，必要时再按 R/G/B 分组 |
| `event_features.py` | 按 Trigger 序号 `% 3` 分组，默认相邻颜色差分 | 主特征只从 `COLOR_RISE` 构建，BLACK 响应单独累计；背景来自颜色亮起前的黑场稳定段 |
| `reconstruction.py` | `log-ls` 求解 `B→R/R→G/G→B`，依赖一次 `黑→R` 锚点 | 删除 v2 下的连续换色方程，直接融合三个独立 `BLACK→Color` 响应，建立公共支持掩膜 |
| `reconstruct_rgb.py` | 报告 RGB Trigger 数和 v1 完整周期 | 报告状态数、BLACK/COLOR 数、三色有效样本、熄灭响应和 v2 协议版本 |
| `config.yaml` | 使用 `rgb_interval_us/expected_rgb_pulses` | 使用 `state_interval_us/expected_state_pulses/expected_color_samples` 等 v2 参数 |

因此本轮不能通过把 `COLOR_RISE` 临时改名为 `RGB_RISE` 来完成适配。这样虽然可能绕过 CSV 读取错误，但会导致周期编号、颜色分组、背景窗口和重建方程继续使用错误语义。

## 3. v2 输入契约与数据模型

### 3.1 CSV 必要字段

首轮实现优先以同名 CSV 为可信语义来源，要求存在：

```text
protocol_version
timestamp_us
id
polarity
role
state_index
cycle_index
phase_index
state
```

必要校验：

1. `protocol_version == rgb-black-sync-v2`；
2. 正式上升沿均为 `id=0, polarity=1`；
3. `state_index` 从 0 连续递增；
4. 每六个状态严格为 `BLACK/R/BLACK/G/BLACK/B`；
5. `cycle_index == state_index // 6`；
6. `phase_index` 在每轮中为 `0,0,1,1,2,2`；
7. BLACK 行使用 `BLACK_RISE`，R/G/B 行使用 `COLOR_RISE`；
8. 默认应得到 150 个状态、25 个完整周期、每色 25 个亮起样本；
9. CSV 的每个正式 Trigger 时间戳都能在 RAW 中按小容差匹配。

### 3.2 内部对象

将当前只表达颜色切换的 `RGBTrigger` 替换或扩展为：

```text
StateTrigger
├── timestamp_us
├── state_index
├── cycle_index
├── phase_index
├── state              # BLACK/R/G/B
└── role               # BLACK_RISE/COLOR_RISE
```

在完成六状态校验后，再构造便于特征提取的：

```text
ColorExposure
├── cycle_index
├── phase_index
├── color
├── baseline_black_trigger
├── color_on_trigger
└── color_off_trigger  # 最后一轮 B 可能没有，允许为空
```

其中：

- `baseline_black_trigger` 是颜色亮起前的 BLACK Trigger，用于确认黑场保持时间和选择背景窗；
- `color_on_trigger` 是构建主通道的 Trigger；
- `color_off_trigger` 是紧随该颜色之后的 BLACK Trigger，用于统计熄灭响应；
- 最后一轮 B 后直接进入 STOP，可能没有正式 BLACK 状态，因此不能为了凑齐熄灭配对而伪造 Trigger。

### 3.3 无 CSV 时的降级规则

v2 第一阶段以 CSV 为必需输入，避免在尚未稳定时依靠位置猜测状态。后续再增加 RAW 恢复：

1. 从 START 和 STOP 标记确定正式段；
2. 按 100 ms 状态间隔恢复全部正式 Trigger；
3. 只有用户明确指定 `--protocol rgb-black-sync-v2` 时，才按六状态序列补充标签；
4. RAW 中无法自行证明照明实际采用 v1 还是 v2，禁止静默猜测协议版本；
5. 降级恢复必须在报告中记录 `metadata_mode=raw_recovery` 和明确警告。

旧 v1 数据是否继续兼容由协议版本分派处理，不能让同一个解析函数同时靠缺失字段猜测两种格式。优先完成 v2 主路径，随后再决定保留独立的 v1 解析器还是提供只读兼容模式。

## 4. 时序窗口与事件分箱

### 4.1 分开估计亮起和熄灭响应

现有代码把所有 Trigger 叠加为一条事件率曲线。v2 至少要分别生成：

- 全部 `COLOR_RISE` 的亮起响应曲线；
- R、G、B 三组 `COLOR_RISE` 响应曲线；
- 全部 `BLACK_RISE` 的熄灭响应曲线。

可以先用全部 `COLOR_RISE` 估计公共响应窗，再检查三种颜色的峰值偏移是否一致。如果某一颜色峰值明显偏离公共窗口，则为该颜色单独估计窗口或给出警告，不能直接把弱响应误判为不存在。

### 4.2 背景窗口

每个主通道样本的背景必须来自该颜色亮起前的 BLACK 稳定段。以 `COLOR_RISE` 为零点，建议先检查：

```text
[-100 ms, -60 ms]  BLACK 切换响应，通常不作为背景
[-50 ms, -10 ms]   BLACK 稳定段，可作为背景候选
[约 -2 ms, +24 ms] BLACK→Color 主响应，具体由样本重新估计
```

实际边界必须由新样本的事件率曲线确定。背景窗不得跨越前一个 BLACK Trigger 的响应，也不得包含当前颜色亮起响应。原先放在 Trigger 后部的安静窗语义需要重写。

### 4.3 显式按窗口归属事件

为每个 `ColorExposure` 分别累计：

- `on_positive/on_negative`：`BLACK→Color` 响应窗；
- `off_positive/off_negative`：`Color→BLACK` 响应窗；
- `black_background_positive/negative`：颜色亮起前的黑场稳定窗。

事件归属应按显式时间区间判断，避免仅用“最近 Trigger”使窗口边界附近的事件被分配到错误状态。所有窗口的上界必须早于下一次状态切换，并在报告中保存最终使用的时间范围。

## 5. 特征、异常过滤与通道融合

### 5.1 主通道特征

对每个颜色亮起样本计算：

```text
S_on = N_pos - N_neg
A_on = N_pos + N_neg
```

对熄灭样本单独计算：

```text
S_off = N_pos - N_neg
A_off = N_pos + N_neg
```

R、G、B 主通道只融合对应的 `S_on/A_on`，绝不把 BLACK Trigger 附近的事件并入下一颜色。`A_off` 和 `S_off` 用于判断照明是否确实熄灭、正负极性是否大体合理以及是否存在残光，不直接加入主 RGB 图。

### 5.2 异常样本过滤

异常检测从原来的“序号 `% 3` 分组”改为按颜色和方向分组。每个样本至少统计：

- 响应事件总量；
- 正负事件比例；
- 峰值偏移；
- 响应空间重心；
- 黑场背景事件率；
- on/off 响应量比值。

使用中位数和 MAD 标记明显异常样本，但必须保存被剔除的 `cycle_index、phase_index、color` 和原因。若某色有效样本少于总样本的 80%，给出降质警告；少于最低完整样本数时停止该通道重建。

### 5.3 跨周期融合

每种颜色默认有 25 个亮起样本。继续采用逐像素中位数或截尾均值融合，并对有效样本数量归一化。三通道共享背景阈值和显示归一化尺度，禁止为了让弱通道“看起来正常”而分别拉满。

公共物体支持掩膜可由三种 `A_on` 的稳健总和或联合置信度生成。现有 `first_dark_to_red` 仅适用于 v1 的一次性黑到红锚点，在 v2 中应替换为三个独立颜色响应形成的公共亮度/支持图。

## 6. v2 重建方法

### 6.1 第一阶段：独立响应基线

将 `response` 作为 v2 首要基线：

1. 分别融合 25 次 `BLACK→R/G/B` 的 `A_on`；
2. 扣除对应黑场背景；
3. 应用热像素掩膜和轻量空间滤波；
4. 使用共享尺度归一化三个通道；
5. 直接组成相对 RGB，并输出三张灰度通道图。

该方法的物理含义是“相对于统一黑场，切换到该颜色时产生的事件响应强度”。它仍不等同于绝对反射亮度，但比 v1 的颜色间差分更容易解释。

### 6.2 有符号响应

可同时保留 `S_on` 作为诊断或第二种方法，用于观察正负极性和估计相对对数亮度变化。由于事件数受对比度阈值、暗电流、环境光和响应饱和影响，不能未经标定就把它当作精确的 `log(I_color/I_black)`。

现有 v1 `log-ls` 的方程：

```text
lR-lB, lG-lR, lB-lG
```

不再适用于 v2，必须在 v2 路径中禁用或替换，不能继续运行后仅修改方法名称。若保留 v1 兼容模式，旧 `log-ls` 只能由 v1 协议分支调用。

### 6.3 标定后处理

新样本完成基础重建后，再使用灰卡和色卡确定：

- 三色 LED/传感器联合通道增益；
- 白平衡参数；
- 3×3 色彩校正矩阵；
- gamma 和显示映射。

当某一颜色响应明显偏弱时，应先查看原始事件量、LED 光功率和传感器光谱响应，不能只在软件中扩大该通道增益。

## 7. 配置与报告升级

`config.yaml` 的协议部分建议改为：

```yaml
protocol:
  version: rgb-black-sync-v2
  channel_id: 0
  state_interval_us: 100000
  state_tolerance_us: 20000
  expected_state_pulses: 150
  expected_complete_cycles: 25
  expected_color_samples: 25
  minimum_complete_cycles: 3
  csv_match_tolerance_us: 50
```

时序配置新增黑场背景窗口、on/off 独立窗口及按颜色诊断开关。参数名称不再沿用容易混淆的 `rgb_interval_us`。

JSON 报告至少增加：

- 输入协议版本和元数据模式；
- 正式状态总数、BLACK/COLOR 上升沿数量；
- 完整六状态周期数；
- R/G/B 原始样本数和有效样本数；
- 三色 `BLACK→Color` 的事件总量、有效像素数和峰值偏移；
- R/G/B 熄灭响应总量及 on/off 比值；
- 黑场背景事件率和热像素数；
- 每个被过滤样本的周期、颜色和原因；
- 最终窗口、归一化方式、校准状态和全部警告。

## 8. 分阶段实施顺序

### M1：协议层适配

修改 `protocol_parser.py`、`config.yaml` 和 `tests/test_protocol.py`：

1. 建立 `StateTrigger` 与 `ColorExposure`；
2. 完成 v2 CSV 读取和九字段校验；
3. 校验 150 个状态及六状态顺序；
4. 将每个 COLOR 与前后 BLACK 正确关联；

6. 完成正常、漏状态、重复状态、错误 role/state/phase、错误协议版本测试。

验收：不读取 RAW/CD 事件也能仅凭合成 CSV 正确得到 25 个完整周期和 R/G/B 各 25 个 `ColorExposure`。

### M2：时序与特征层适配

修改 `timing.py`、`event_features.py` 及其测试：

1. COLOR 与 BLACK 响应曲线分离；
2. 黑场稳定背景窗重新定义；
3. 主通道只累计 `COLOR_RISE`；
4. BLACK 熄灭响应单独累计；
5. 异常过滤按颜色与方向分组；
6. 用合成事件验证窗口之间不串扰。

验收：人工放入 BLACK 窗、COLOR 窗和边界附近的事件后，各类计数严格进入预期数组。

### M3：重建流程、输出与兼容策略

修改 `reconstruction.py`、`reconstruct_rgb.py`、`README.md` 和报告：

1. v2 默认使用独立响应基线；
2. 替换 `first_dark_to_red` 和 v1 连续差分假设；
3. 更新 CLI 描述、进度输出和 JSON 字段；
4. 根据协议版本显式分派 v1/v2，或先对 v1 给出明确“不支持”提示；
5. 保持四宫格尺寸、方向和共享归一化不变。

验收：全部零事件、弱 B 通道、单个热像素和异常样本都不会产生 NaN、越界或静默算法降级。

### M4：真实 v2 样本验证

使用本次新采集的同名 RAW/CSV：

1. 检查 CSV 与 RAW Trigger 匹配率；
2. 输出三色独立时序曲线和 BLACK 熄灭曲线；
3. 重建 R/G/B/RGB 四宫格；
4. 分别报告每色事件总量、有效像素数、峰值偏移和保留样本数；
5. 与 7 月 17 日 v1 样本的通道统计和空间可见性进行对比；
6. 保存结果、报告和必要的中间特征。

验收重点不是要求三通道强度完全一致，而是灰卡或同一物体应在三个通道的相同空间位置出现。若 B 仍明显偏弱，应转入 LED 光功率、环境光和传感器光谱响应排查，不能继续用归一化掩盖。

## 9. 测试清单

### 协议测试

- 合法的 25×6 状态 CSV；
- 少一个 BLACK、少一个 COLOR、重复 Trigger；
- `state_index/cycle_index/phase_index` 不一致；
- BLACK 使用 `COLOR_RISE` 或颜色使用 `BLACK_RISE`；
- 错误通道、错误极性、错误协议版本；
- CSV 与 RAW 时间戳不匹配；
- 最后一轮 B 无正式 off BLACK 的合法边界。

### 时序与特征测试

- 三色具有不同峰值偏移；
- BLACK 熄灭响应不会进入 COLOR 主通道；
- 黑场背景窗不包含 BLACK 切换响应；
- Trigger 前负偏移事件能够正确归属；
- 热像素、零事件和窗口边界事件；
- 某一颜色的单个异常周期只影响对应颜色；
- 弱 B 响应保留其真实相对强度，不被独立拉满。

### 输出与回归测试

- 四格尺寸和方向完全一致；
- 同一输入重复运行结果一致；
- 报告中的计数与内部对象一致；
- v2 路径不调用 v1 `log-ls` 方程；
- 若保留 v1，现有 9 项测试继续通过，并增加协议分派测试。

## 10. 风险与处置

| 风险 | 可能表现 | 处置 |
|---|---|---|
| 只改 CSV 字段 | 程序能运行但颜色分箱仍错误 | 以 `ColorExposure` 贯穿时序、特征和重建层 |
| BLACK 环境光不够暗 | 三色亮起事件都偏少 | 报告黑场背景；检查遮光和 LED 对比度 |
| BLACK 切换响应拖尾 | 污染下一颜色背景窗 | 根据实测曲线后移背景窗或延长状态保持时间 |
| 三色响应延迟不同 | 公共窗口漏掉弱通道 | 输出分色时序曲线，必要时使用分色窗口 |
| 最后一轮 B 无 off BLACK | 熄灭样本少一个 | 允许 off 为空，报告实际 off 样本数 |
| v1/v2 自动误判 | 得到看似正常但语义错误的图 | 以 `protocol_version` 显式分派，缺失时要求用户指定 |
| 每色样本从 50 降至 25 | 稳健融合统计量减少 | 使用中位数/MAD并报告置信度；需要时增加采集周期 |
| 单强度照明仍无法恢复绝对亮度 | RGB 与普通相机颜色不一致 | 明确标为未标定相对响应，后续增加多亮度和色卡标定 |
