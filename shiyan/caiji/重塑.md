# GenX320 RGB 同步采集重塑计划

## 1. 本次审查结论

两个脚本均通过 Python 语法检查，但当前采集流程存在一个确定的问题和若干同步风险。

### `sync_led_trig.py`

当前脚本确实按照 `R -> G -> B -> R` 循环切换颜色，并且每次执行 `pixels.show()` 时都会产生一次 Trigger 高脉冲：

1. GPIO 17 拉高；
2. 调用 `pixels.show()`，向整块 256 像素软屏发送数据；
3. GPIO 17 拉低；
4. 等待至下一个 0.1 s 周期。

所以“每次颜色切换都有 Trigger”这一逻辑成立，初始颜色是红色，理论频率为 10 Hz，15 s 大约产生 150 个脉冲。

但是当前 Trigger 并不精确代表“新颜色已经生效”的瞬间。上升沿发生在 `pixels.show()` 之前，下降沿发生在其返回之后。WS2812 数据需要发送并锁存，且即使只点亮 16 个灯，代码仍会发送全部 256 个像素的数据，因此高电平宽度取决于 SPI/驱动执行时间，而不是一个明确、固定的脉宽。新颜色真正生效的时刻更接近 `pixels.show()` 结束处。

还需注意：

- `pixel_order=GRB` 通常是对 WS2812 物理通道顺序的适配，传入的 `(255, 0, 0)` 仍按 RGB 使用；但最终颜色必须用实物确认一次。
- 代码假定软屏是普通逐行映射。若实际软屏为蛇形布线，当前 16 个索引不一定形成物理中心 4x4。
- 当前脉冲宽度由 `pixels.show()` 偶然决定，应改成明确的同步时序，并用示波器或逻辑分析仪测量 GPIO 17 与 LED DIN/光电响应的相对时间。
- 发送周期采用累计目标时间，基本具备漂移补偿；但当一次循环超过 `INTERVAL` 时，没有记录超期或跳周期，会连续追赶执行。

### `receive_record.py`

当前“没有运行发送端也会生成并写入 RAW”不是相机误触发，而是录制调用位置造成的：

```python
events_stream.log_raw_data(save_path)
```

在等待 Trigger 之前就已执行。之后的 `trigger_detected` 只决定何时开始 15 s 倒计时，并没有决定何时开始写文件。因此 RAW 实际内容是：

```text
接收脚本启动 -> 任意长的等待期 -> 首个上升沿 -> 再录 15 s
```

若发送端始终不启动，程序会一直等待，RAW 也会持续增长，直至用户中断。

此外还有以下问题：

- 任意一个上升沿就会被当成有效启动，缺少通道 ID、连续脉冲间隔和事件量验证，容易被干扰误启动。
- `get_i_trigger_in()` 不存在或 `enable(0)` 失败时，程序仍继续运行；这种情况应该立即报错退出。
- 没有等待超时、最大文件保护、失败文件删除/改名策略。
- `time.time()` 会受系统时间调整影响，持续时间应使用 `time.monotonic()`；真正分段时应优先使用相机时间戳。
- `log_raw_data()` 的返回值未检查，文件打开失败仍可能显示录制成功。
- 校验函数把上升沿和下降沿一起累计，所以“触发点数”通常约为脉冲数的两倍；它也没有校验 `id`、极性、相邻间隔、RGB 周期数和文件时长。
- 输出目录相对当前终端目录，而不是脚本目录，可能导致文件保存位置不稳定。

## 2. 推荐的可靠采集协议

不能简单地把 `log_raw_data()` 移到“检测到首个 Trigger”之后：Trigger 被迭代器解码出来时，该边沿已经过去，RAW 录制无法倒退包含它。可靠方案是给发送端增加“握手前导”，让接收端先确认发送程序真的启动，再在正式 RGB 数据开始前打开最终 RAW。

推荐时序：

```text
接收端进入 ARMED 状态
        |
发送端输出特征前导脉冲/预热颜色序列
        |
接收端确认：脉冲数量 + 间隔 + 通道 ID（可选再加事件率）
        |
接收端开启最终 RAW 录制
        |
200~500 ms 保护间隔
        |
发送端发出明确 START 标记
        |
正式 R -> G -> B 周期
        |
明确 STOP 标记，停止并校验
```

前导必须与正式的 10 Hz 单脉冲序列明显不同，例如 3 个间隔 30 ms 的短脉冲，随后保持 300 ms 安静期。接收端只有识别完整模式才进入录制，单个毛刺不会启动。正式 RGB 段仍使用 100 ms 周期。

由于两个脚本看起来运行在同一台树莓派，还可额外增加本机 Unix socket/临时状态文件作为“接收端已就绪”握手，但它只能作为辅助；最终同步基准仍应以相机记录到的硬件 Trigger 为准。更彻底的后续方案是由一个总控程序统一启动接收与 LED 子模块。

## 3. 分阶段实施计划

### 阶段 A：先测量并固定 Trigger 与 LED 的关系

- 将 Trigger 改为固定脉宽，例如 1~5 ms，不再让脉宽取决于 `pixels.show()`。
- 建议先执行 `pixels.show()`，待函数返回后立即产生固定 Trigger 上升沿，使 Trigger 表示“新颜色已提交”；实际光学延迟再通过测量标定。
- 每次切换记录 `cycle_index`、颜色、计划时刻、实际单调时刻、超期量。
- 检查 16 个灯的物理位置及红绿蓝通道顺序。
- 用示波器/逻辑分析仪，最好再加光电二极管，测量新颜色生效与 Trigger 的延迟和抖动；把最终偏移写入分析参数，而不是凭代码注释假定为零。

验收：连续至少 300 次切换均有成对边沿；周期接近 100 ms；固定脉宽和 LED/Trigger 相对延迟的波动满足实验需求。

### 阶段 B：把接收端改成状态机

建议状态如下：

```text
INIT -> ARMED -> PREAMBLE_CONFIRMED -> RECORDING -> VERIFY -> DONE
                \-> TIMEOUT / INVALID
```

- `INIT`：初始化设备、确认 Events Stream 和 Trigger In 存在；检查 `enable(channel_id)` 成功。
- `ARMED`：只消费和分析实时事件，不调用 `log_raw_data()`；清空启动前遗留 Trigger。
- 前导确认条件：指定 `id` 的连续上升沿数量、间隔容差、极性均符合协议。
- 可选事件量门限：在每次颜色切换附近统计 CD 事件数或 events/s，作为第二重条件。它不应单独作为启动条件，因为环境运动、工频闪烁也可能产生大量事件。
- `PREAMBLE_CONFIRMED`：生成最终文件名，检查 `log_raw_data()` 返回成功，然后通知/等待发送端进入保护间隔。
- `RECORDING`：以 START/STOP 标记或相机时间戳控制正式段；墙钟只作为故障兜底。
- `TIMEOUT`：例如 30 s 内未识别有效前导就退出，不创建正式 RAW。
- `finally`：只停止一次录制，明确保存成功、失败或被中断；无效临时文件使用 `.partial` 后缀并按策略删除或保留诊断。

验收：只运行接收脚本超过等待超时，不产生 `.raw`；发送单个毛刺不录制；运行发送脚本后只产生一个有效 RAW。

### 阶段 C：定义可供重建使用的 RGB 元数据

- 固定约定 START 后第一个颜色一定为 R，之后严格为 G、B。
- 对每个 Trigger 保存 `timestamp_us, id, polarity, cycle_index, color` 到同名 CSV/JSON sidecar。
- 校验上升沿和下降沿分别计数，不混称为“脉冲数”。
- 校验相邻同极性边沿的周期，报告丢脉冲、重复脉冲、错误通道及超容差周期。
- 推荐用 STOP 标记结束，而不是仅靠“启动后再等 15 s”，避免发送端提前退出时仍录背景事件。
- 若 STOP 丢失，使用 `TOTAL_DURATION + 容差` 作为硬超时。

验收：15 s、10 Hz 时，正式段预期约 150 次颜色切换、约 50 个完整 RGB 周期；实际数量以 START/STOP 边界和时间戳校验结果为准。

### 阶段 D：录制质量和故障保护

- 输出路径改为相对 `receive_record.py` 所在目录，并建立 `trigger_records/`。
- 文件先写为 `.partial.raw`，完成完整校验后原子改名为 `.raw`。
- 设置等待超时、录制硬超时和最小/最大事件数保护。
- 记录 SDK 版本、相机序列号、分辨率、bias、ROI、LED 亮度、周期、前导协议版本等实验参数。
- 在实际相机上做四组测试：无发送端、单毛刺、正常完整发送、发送中途异常退出。

## 4. 建议的实现优先级

1. 首先修复接收端“等待期已经写 RAW”的确定性缺陷，并加入等待超时。
2. 同时加入发送端前导协议，否则把录制调用后移会丢失用于对齐的首个边沿。
3. 固定 Trigger 脉宽和 LED/Trigger 的先后关系，并完成硬件测量。
4. 增强触发序列、事件量、START/STOP 和输出文件校验。
5. 最后再依据可靠的 Trigger 时间戳进行 RGB 事件分箱和真实图像重建。

## 5. 不建议采用的简单修补

- **仅把 `log_raw_data()` 移到首个上升沿之后**：会丢首个同步边沿和它之前已发生的颜色变化。
- **仅以“事件很多”作为启动条件**：物体运动、传感器噪声和灯光闪烁都可能误触发。
- **继续先录一个无限增长的预录文件**：虽然能保留首个边沿，但无法满足“发送端未启动时不录 RAW”，也存在磁盘占满风险。
- **只检测单个 Trigger 上升沿**：抗干扰能力不足，无法证明是 RGB 控制脚本启动。

## 6. 实施前需要在树莓派上确认的信息

- Metavision SDK/驱动的确切版本。
- GenX320 板卡型号及 J3 Trigger In 的电平、边沿和通道映射要求。
- `enable(0)` 在当前设备上的返回值以及 RAW 中实际记录的 Trigger `id`。
- `pixels.show()` 的实测耗时、软屏是否蛇形布线、R/G/B 显示是否正确。
- 正常 RGB 切换时每个 50 ms/100 ms 时间窗的大致事件数，用于设置可选事件率门限。

## 8. 首批真实数据重建后的反思（2026-07-17）

首批同名 RAW/CSV 已经能够通过协议校验并完成四通道重建：

- RAW 中识别到 150 个正式 RGB Trigger，共 50 个完整 RGB 周期；
- 自动估计的光学响应峰相对 Trigger 约为 `+4.5 ms`；
- 实际响应窗口为 `[-2 ms, +24 ms]`；
- 经异常周期过滤后，R、G、B 分别保留 41、44、44 个有效周期，协议和同步本身没有报告警告。

但是重建暴露出当前 `R -> G -> B` 连续换色协议的根本局限。分析端的三个通道并不是普通相机意义上的独立 R、G、B 亮度，而是相邻颜色切换产生的事件响应：

```text
B -> R：当前 R-Channel 的主要来源
R -> G：当前 G-Channel 的主要来源
G -> B：当前 B-Channel 的主要来源
```

事件相机只在对数亮度发生足够变化时输出事件。如果物体在绿光和蓝光下的亮度相近，`G -> B` 就可能几乎不产生事件；这并不等价于物体不存在，也不一定等价于物体完全不反射蓝光。

本次融合后的实测统计为：

| 通道 | 有响应像素数 | 融合响应总量 |
|---|---:|---:|
| R | 507 | 460.49 |
| G | 477 | 416.04 |
| B | 88 | 77.62 |

B 通道只有 R/G 通道约六分之一的稳定响应，因此同一物体在 B-Channel 中基本不可见。B 仍保留了 44 个有效周期，与 G 相同，所以主要原因不是 B 周期被异常过滤器大量删除，而是 `G -> B` 转换本身没有在该物体区域产生足够稳定的事件。蓝色 LED 实际光功率、物体的蓝光反射率、GenX320 的光谱响应和环境光也可能进一步削弱蓝色响应，需要用灰卡和光电测量分别排查。

### 8.1 下一版采集协议：在各颜色之间插入黑场

正式照明序列建议由：

```text
R -> G -> B -> R -> ...
```

改成：

```text
BLACK -> R -> BLACK -> G -> BLACK -> B -> ...
```

这样三个颜色通道分别由同一种基准到目标颜色的变化构建：

```text
BLACK -> R：R-Channel
BLACK -> G：G-Channel
BLACK -> B：B-Channel
```

该方案消除了“前一个颜色”对当前通道的耦合，使 R、G、B 都相对于统一黑场测量，更容易在三个单通道中得到空间对应的同一物体。颜色熄灭产生的 `R/G/B -> BLACK` 负事件也应保留，可作为独立的下降沿响应和质量校验，但不能与下一次 `BLACK -> Color` 的上升响应混在同一个窗口中。

### 8.2 Trigger 和元数据需要同步升级

- 每次 `pixels.show()` 返回后仍发送固定脉宽 Trigger，包括切换到 BLACK 和切换到 R/G/B；
- CSV 的每一行必须明确记录目标状态，至少包含 `timestamp_us, cycle_index, state, role`；
- `state` 建议使用 `BLACK/R/G/B`，`role` 建议区分 `BLACK_RISE` 与 `COLOR_RISE`；
- 分析端只用 `BLACK -> R/G/B` 的 `COLOR_RISE` 响应构建三个主通道，同时单独统计颜色熄灭响应；
- START、STOP 和 PREAMBLE 规则继续保留，且正式段的预期 Trigger 数量需要按新序列重新计算；
- 黑场保持时间和颜色保持时间必须足够覆盖光学响应窗口，并在下一次切换前留下安静段用于背景估计；
- 建议先保持每个状态 100 ms。这样一个完整 `BLACK/R/BLACK/G/BLACK/B` 周期约为 600 ms；若实验必须维持原有总时长，应重新计算完整周期数，不能继续沿用“15 s 等于 50 个 RGB 周期”的旧假设。

### 8.3 新协议的验收标准

1. 灰卡或白色物体在 R、G、B 三个通道中应出现在相同空间位置，不能再出现某通道整体缺失而有效周期数正常的情况。
2. 分别报告三种 `BLACK -> Color` 的事件总量、有效像素数、峰值偏移和保留周期数。
3. 若某颜色响应仍显著低于另外两色，应优先测量对应 LED 的实际光功率，再判断是否调整亮度；不能仅在分析端把该通道单独拉满。
4. 检查 BLACK 状态是否真正接近无光，并确认环境光不会使黑场与颜色照明之间的对数亮度差过小。
5. 用光电二极管分别测量 R、G、B 和 BLACK 的光输出，确认 Trigger 后的响应偏移、峰值和衰减均落在分析窗口内。
6. 保存一组灰卡、一组标准色卡和一组无物体黑场数据，供白平衡、色彩矩阵、热像素及背景阈值标定。

插入黑场能够解决当前最明显的通道耦合问题，但事件相机仍然不直接测量绝对亮度。若后续目标是接近普通相机的真实 RGB，而不只是得到稳定的相对色彩/伪彩结果，还应为每种颜色增加多个亮度等级，并配合灰卡或色卡完成标定。
